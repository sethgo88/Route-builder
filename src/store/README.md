# store/routeStore.ts

Single Zustand store that holds all mutable app state. Every component that reads or writes app data goes through here.

---

## State shape

```ts
// Waypoints the user has placed on the map
waypoints: Waypoint[]

// Computed route returned by GraphHopper (GeoJSON Feature<LineString>)
route: Feature<LineString> | null

// [distanceKm, elevationM] pairs extracted from the route's 3D coordinates
elevationData: [number, number][]

// Aggregated stats derived from the GraphHopper response
routeStats: RouteStats | null   // { distanceKm, gainM, lossM }

// When true, routing uses the 'hike' profile (follows OSM trails)
// When false, uses 'foot' (more direct, less trail-constrained)
isSnapping: boolean

// True while a GraphHopper fetch is in flight — drives the loading indicator
isLoading: boolean

// Written by ElevationProfile when the user taps the chart;
// read by RouteMap to trigger a camera fly-to
focusCoordinate: [number, number] | null
```

---

## Types

### `Waypoint`

```ts
interface Waypoint {
  id: string          // Unique ID generated by makeId() — used as React key and drag handle
  coordinate: Coordinate
}

interface Coordinate {
  longitude: number
  latitude: number
}
```

Coordinates are stored as named `{ longitude, latitude }` objects rather than a `[lng, lat]` tuple. The named shape is used throughout the store and services. When MapLibre needs a coordinate, it always expects a `[lng, lat]` tuple — the conversion happens at the call site (`[wp.coordinate.longitude, wp.coordinate.latitude]`). This explicit conversion makes the GeoJSON order visible rather than hiding it behind an implicit index.

### `RouteStats`

```ts
interface RouteStats {
  distanceKm: number   // Total route distance in kilometres
  gainM: number        // Total elevation gain in metres (from GraphHopper's 'ascend')
  lossM: number        // Total elevation loss in metres (from GraphHopper's 'descend')
}
```

## Actions

| Action | What it does |
|---|---|
| `addWaypoint(coord)` | Appends a new waypoint with a generated ID |
| `moveWaypoint(id, coord)` | Updates a single waypoint's position (called on drag end) |
| `removeWaypoint(id)` | Filters out a waypoint by ID (called on marker tap) |
| `undoLastWaypoint()` | Removes the most recently added waypoint (slice off last element) |
| `setRoute(route)` | Stores the GeoJSON LineString from GraphHopper |
| `setElevationData(data)` | Stores the `[distKm, elevM]` pairs for the chart |
| `setRouteStats(stats)` | Stores distance/gain/loss summary |
| `setIsSnapping(value)` | Toggles the routing profile; `useRouting` re-fetches when this changes |
| `setIsLoading(value)` | Shown as a spinner in `RouteMap` |
| `setFocusCoordinate(coord)` | Triggers a camera fly-to from `ElevationProfile` → `RouteMap` |
| `clearAll()` | Resets waypoints, route, elevation data, and focus coordinate |
| `loadWaypoints(coords)` | Replaces all waypoints (used by GPX import) |

---

## Why the state is split into separate `set*` actions

Route data (`route`, `elevationData`, `routeStats`) is written by the async routing hook, not by user gestures. Giving each field its own setter keeps the store simple and lets the hook update fields independently as each step of the async pipeline completes. An alternative would be a single `setRouteResult(result)` action — either works, but individual setters make it easier to clear just the route without clearing stats.

---

## Why `focusCoordinate` lives in the store

`ElevationProfile` and `RouteMap` are siblings in the component tree — neither is an ancestor of the other. The elevation chart needs to tell the map camera where to fly. Lifting state up to a common parent (`ControlsPanel`) would mean passing a ref or callback through several layers. The store is already shared global state, so `focusCoordinate` is the cleanest channel between two otherwise unrelated components.

## ID generation

```ts
function makeId(): string {
  return `wp-${Date.now()}-${Math.floor(Math.random() * 10000)}`;
}
```

IDs do not need to be globally unique across sessions — they just need to be stable for the lifetime of a single route-building session. `Date.now()` combined with a random suffix is collision-resistant enough for this use case and avoids a UUID dependency.
